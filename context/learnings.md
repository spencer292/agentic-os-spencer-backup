### tool-jobber
- 2026-07-06: Invoice cash-flow modelling. Key fields: `issuedDate` / `dueDate` / `receivedDate`, `invoiceStatus` (draft/sent_not_due/awaiting_payment/past_due/paid/bad_debt), `amounts { total invoiceBalance paymentsTotal depositAmount }`. Filter `InvoiceFilterAttributes` supports `status`, `issuedDate:{after,before}` (Iso8601DateTimeRangeInput uses `after`/`before`/`eq`), sort `{key:ISSUED_DATE, direction:DESCENDING}`. Account had 14,412 invoices; 12-mo history = 5,427 (~108 pages @ 50).
- 2026-07-06: **Deposits (new customers only):** a new-customer quick-fix job is ONE $450 invoice with `depositAmount:150`. The $150 deposit is collected at booking and is NOT in `paymentsTotal` or `invoiceBalance` (balance already nets it out). So `depositAmount>0` is a reliable new-customer marker, and any cash model must add deposits as up-front cash or it under-counts collections (~$58,900/yr here; aged months read ~100% collected once deposits are credited). Existing customers = no deposit, billed at end of service ($150 no-catch bill or full rate).
- 2026-07-06: **Token rotation gotcha** — rapid parallel `jobber-api.mjs` calls can desync the rotating refresh token → HTTP 401 "Token request failed" on the token endpoint. Fix is re-auth (`jobber-api.mjs auth`, browser). Run Jobber calls sequentially, not in parallel bursts.
- 2026-07-06: **Job Custom Fields are writable WITHOUT the custom-fields scope.** The app lacks the "Custom Field Configurations" scope (reading/creating field *configs* returns "hidden due to permissions"), BUT field VALUES write fine via `jobEdit(jobId, {customFields:[{id, valueDropdown|valueNumeric|valueText|valueTrueFalse}]})` under Jobs:write. Get each field's per-job `id` by reading `job.customFields` (union: CustomFieldText/Numeric/Dropdown/TrueFalse — request `id`+`label`, but NOT `customFieldConfiguration` which is scope-blocked). Match fields by TRIMMED label (names can have trailing spaces). **Dropdown values must EXACTLY equal an option** — a "None " option with a trailing space rejected `valueDropdown:"None"` ("dropdown value must be one of the available options"); have the user clean the option.
- 2026-07-06: **Turning OFF refresh-token rotation INVALIDATES the current refresh token.** After the user flipped rotation off in the Developer Center, the existing `JOBBER_REFRESH_TOKEN` died — both the CLI and n8n got 401 "refresh token is not valid." Fix: re-auth (`jobber-api.mjs auth`) to mint a fresh token; with rotation off that new token is permanent/stable and reusable by multiple consumers.
- 2026-07-06: **Jobber auto-allows `localhost` OAuth callbacks** — you cannot list `http://localhost:8734/callback` in the app's Callback URL field (Developer Center rejects it: "'localhost' is supported automatically and shouldn't be listed"). So the CLI's localhost re-auth works even when the app's configured callback is a remote URL (e.g. n8n's). No callback change needed to re-auth the CLI.
- 2026-07-06: **Visit scheduling from notes** — `visitEditSchedule(id, {startAt:{date,timezone},endAt})` moves a visit; `visitCreate(jobId, {visits:[{title, schedule:{startAt,endAt,teamMemberIdsToAssign}}]})` adds one (payload field is `createdVisits`, not `visits`). Read upcoming visits via `job.visits(first,filter:{status:UPCOMING})`. Rule: same calendar month → pull earlier; later month → leave + add (never pull a future-month visit back).
- 2026-07-09: **Batch write-back of visit times (90 visits, 0 fail).** `visitEditSchedule(id, {startAt:{date,time,timezone}, endAt:{date,time,timezone}})` with `timezone:"America/Los_Angeles"` + local `time:"HH:MM:00"` sets an exact time; the mutation returns `visit{ startAt endAt }` — read it back (`...T06:48:00-07:00`) as inline verification. Setting startAt also reorders the board (Jobber sorts by start time). Window convention here = start +3h. **Throttle:** heavy visit reads cost-throttle fast (`"Throttled"`/`THROTTLED`) and rapid retries drain the bucket further — paginate day visits at 25/page with escalating backoff; filter a Pacific day by UTC window `${date}T07:00:00Z`..`${next}T07:00:00Z`. Schema gotchas: `Visit.client.name` is a scalar (not `{full}`); `PropertyAddress` has street/city/province/postalCode but **no** latitude/longitude; `Visit.duration`/`endAt` reflect the booked window (120min/all-day defaults), NOT real service time — don't derive service duration from them (OR uses a fixed 10-min stop for Got Moles).

- 2026-07-06: **httpRequest v4.4 empty-body gotcha (n8n Cloud 2.27).** A `jsonBody` set to `={{ JSON.stringify({query:'... GraphQL with many braces ...'}) }}` sent an EMPTY body `{}` → Jobber 400 "Invalid query." Fix: for a STATIC query, set `jsonBody` to a literal JSON string (no `=` expression). ## General

- 2026-07-06: **Do not flip a live-data-writing automation to production until the WHOLE workflow is built and validated.** The Got Moles route automation shipped only the OptimoRoute→Jobber write-back (missing the Jobber→OptimoRoute ingest + re-optimize, and tech-assignment handling) plus a bad 7am–7pm placeholder step. Going live scrambled the real Jobber board (visits moved to wrong days, blanket placeholder times). "0 errors" on a run is NOT proof of correctness — verify what a real run actually CHANGES against a real day, and make the user review it before go-live. Go-live is a hard gate, not a flip.
