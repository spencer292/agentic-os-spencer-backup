### tool-jobber
- 2026-07-06: Invoice cash-flow modelling. Key fields: `issuedDate` / `dueDate` / `receivedDate`, `invoiceStatus` (draft/sent_not_due/awaiting_payment/past_due/paid/bad_debt), `amounts { total invoiceBalance paymentsTotal depositAmount }`. Filter `InvoiceFilterAttributes` supports `status`, `issuedDate:{after,before}` (Iso8601DateTimeRangeInput uses `after`/`before`/`eq`), sort `{key:ISSUED_DATE, direction:DESCENDING}`. Account had 14,412 invoices; 12-mo history = 5,427 (~108 pages @ 50).
- 2026-07-06: **Deposits (new customers only):** a new-customer quick-fix job is ONE $450 invoice with `depositAmount:150`. The $150 deposit is collected at booking and is NOT in `paymentsTotal` or `invoiceBalance` (balance already nets it out). So `depositAmount>0` is a reliable new-customer marker, and any cash model must add deposits as up-front cash or it under-counts collections (~$58,900/yr here; aged months read ~100% collected once deposits are credited). Existing customers = no deposit, billed at end of service ($150 no-catch bill or full rate).
- 2026-07-06: **Token rotation gotcha** — rapid parallel `jobber-api.mjs` calls can desync the rotating refresh token → HTTP 401 "Token request failed" on the token endpoint. Fix is re-auth (`jobber-api.mjs auth`, browser). Run Jobber calls sequentially, not in parallel bursts.
- 2026-07-06: **Job Custom Fields are writable WITHOUT the custom-fields scope.** The app lacks the "Custom Field Configurations" scope (reading/creating field *configs* returns "hidden due to permissions"), BUT field VALUES write fine via `jobEdit(jobId, {customFields:[{id, valueDropdown|valueNumeric|valueText|valueTrueFalse}]})` under Jobs:write. Get each field's per-job `id` by reading `job.customFields` (union: CustomFieldText/Numeric/Dropdown/TrueFalse — request `id`+`label`, but NOT `customFieldConfiguration` which is scope-blocked). Match fields by TRIMMED label (names can have trailing spaces). **Dropdown values must EXACTLY equal an option** — a "None " option with a trailing space rejected `valueDropdown:"None"` ("dropdown value must be one of the available options"); have the user clean the option.
- 2026-07-06: **Turning OFF refresh-token rotation INVALIDATES the current refresh token.** After the user flipped rotation off in the Developer Center, the existing `JOBBER_REFRESH_TOKEN` died — both the CLI and n8n got 401 "refresh token is not valid." Fix: re-auth (`jobber-api.mjs auth`) to mint a fresh token; with rotation off that new token is permanent/stable and reusable by multiple consumers.
- 2026-07-06: **Jobber auto-allows `localhost` OAuth callbacks** — you cannot list `http://localhost:8734/callback` in the app's Callback URL field (Developer Center rejects it: "'localhost' is supported automatically and shouldn't be listed"). So the CLI's localhost re-auth works even when the app's configured callback is a remote URL (e.g. n8n's). No callback change needed to re-auth the CLI.
- 2026-07-06: **Visit scheduling from notes** — `visitEditSchedule(id, {startAt:{date,timezone},endAt})` moves a visit; `visitCreate(jobId, {visits:[{title, schedule:{startAt,endAt,teamMemberIdsToAssign}}]})` adds one (payload field is `createdVisits`, not `visits`). Read upcoming visits via `job.visits(first,filter:{status:UPCOMING})`. Rule: same calendar month → pull earlier; later month → leave + add (never pull a future-month visit back).
- 2026-07-09: **Batch write-back of visit times (90 visits, 0 fail).** `visitEditSchedule(id, {startAt:{date,time,timezone}, endAt:{date,time,timezone}})` with `timezone:"America/Los_Angeles"` + local `time:"HH:MM:00"` sets an exact time; the mutation returns `visit{ startAt endAt }` — read it back (`...T06:48:00-07:00`) as inline verification. Setting startAt also reorders the board (Jobber sorts by start time). Window convention here = start +3h. **Throttle:** heavy visit reads cost-throttle fast (`"Throttled"`/`THROTTLED`) and rapid retries drain the bucket further — paginate day visits at 25/page with escalating backoff; filter a Pacific day by UTC window `${date}T07:00:00Z`..`${next}T07:00:00Z`. Schema gotchas: `Visit.client.name` is a scalar (not `{full}`); `PropertyAddress` has street/city/province/postalCode but **no** latitude/longitude; `Visit.duration`/`endAt` reflect the booked window (120min/all-day defaults), NOT real service time — don't derive service duration from them (OR uses a fixed 10-min stop for Got Moles).

- 2026-07-10: **`jobs(first: N)` default order is a coverage TRAP for day-based work.** The notes-automation read relied on the first 40-50 jobs happening to include the day's visited jobs — on 07-09 it found 0 of 65; even the "proven" 07-06 sync got 31 of 85. Correct pattern: **visits-first** — page `visits(filter:{startAt:{after,before}})` for the PT day (cheap), dedup to `job.id`s, then fetch heavy per-job data (notes/custom fields) via ALIASED batch queries `j0: job(id:..){...} j1: ...` at ~15 jobs/call. Full coverage, and stays under the cost throttle (a single `jobs(first:100)` pull with 40 notes each = instant THROTTLED).
- 2026-07-10: **`visits(filter:{status:UPCOMING})` is the wrong check for "did the follow-up get scheduled"** — a visit happening today (in-window) or already completed is not UPCOMING and reads as missing (false-flagged Mike Coile, whose follow-up was literally in progress). Use a DATE filter instead: `visits(filter:{startAt:{after:"<end of noted day UTC>"}})`. Also: **closed programs are not cadence gaps** — "The Quick Fix — One-Month Mole Control Program" is a RECURRING job that legitimately ENDS (5 weeks); when it's done Jobber sets `jobStatus: archived`. Cadence-gap checks must exclude archived jobs; a NON-archived job with nothing upcoming is real signal (`requires_invoicing` = serviced but never billed — caught a live one). `jobStatus` values seen: archived, requires_invoicing, action_required. `lineItems{nodes{name}}` identifies the program; job GraphQL id = base64 of `gid://Jobber/Job/<work_orders URL number>`.
- 2026-07-10: **`job.notes` returns OLDEST-first — always use `notes(last: N)` for recent notes.** `notes(first: 40)` on a 40+-note job (the oldest clients) reads 2024 history; the "latest by createdAt" reduce then silently picks an ancient note, making long-history jobs invisible to the notes automation (Chris Doll #5369: 61 notes, current "Add visit" note missed — Spencer caught it). `last:` returns the newest N (still ascending); request `totalCount` when diagnosing. Same-day parser trap: a tech consistently writes **"Ad visit"** for "Add visit" — parse with `/\bad{1,2}\s+visit/i`; shorthand parsers need misspelling tolerance + an exceptions lane, not exact phrases.
- 2026-07-10: **Client tags:** filter is `clients(filter:{tags:["<exact label>"]})` — label must match EXACTLY (the Got Moles program tag is `"TMCP - Active"`, not "TMCP", which returns 0). Read via `client.tags(first:N){nodes{label}}`. Connections expose `totalCount` (cheap way to size a sweep first). **Recurring-job identity lives in lineItems, not title** — 4,646 of 4,749 recurring jobs have a blank title; the program is the line item name (`Total Mole Control Program -- Year round protection` / `Total Mole Control Package` = TMCP; also barter/warranty/friends-family variants that ARE program work billed differently). Full shallow sweep of 4,749 jobs at 50/page (id/title/status/client/property) throttles every ~2,000 rows — 5s backoff recovers; ~5 min total. Aliased 15-job batches for lineItems on 698 jobs ran clean at 400ms spacing.
- 2026-07-10: **Tag writes:** `clientEdit(clientId, input:{tagsToAdd:["label"]})` is ADDITIVE (existing tags kept — verified; `tagsToDelete` is the inverse). Return `client{tags{nodes{label}}}` for inline verification. **Job pricing:** `job.total` = price per billing period; cadence in `invoiceSchedule{billingFrequency scheduleSummary}` (PERIODIC "Monthly on the last day of the month" is the TMCP standard; also Yearly / Every 90 days / ON_COMPLETION / NEVER "Don't remind me to invoice"). `invoices{totalCount}` + `invoicedTotal` on the job distinguish real comps from billing mistakes ($0-line jobs that still invoice = manually billed). The CLI must run from the repo root — it resolves `.env` from cwd.
- 2026-07-12: **`visitEditSchedule` time strings must be exact `HH:MM:SS`** — OptimoRoute's `scheduledAtDt` ("2026-07-14 07:36:32") already carries seconds; appending ":00" produced "07:36:32:00" and Jobber's validation rejected ALL 416 writes atomically (harmless — nothing partial). - 2026-07-10: **Jobber returns partial "hidden due to permissions" errors alongside good data** (e.g. `RequestNote` objects inside `job.notes` when the app lacks Requests scope) — HTTP 200, `data` populated, hidden nodes come back as null in the list. The `jobber-api.mjs` CLI treats ANY `errors` array as fatal and discards the data, so scripts touching notes must use direct fetch and filter: fail only on errors NOT matching `/hidden due to permissions/`, and skip null note nodes.

## General

- 2026-07-19: **Never "dry-check" a mutating script by trimming its output** — `script.mjs | head -2` still EXECUTES everything; only the printout is cut.
