### Phase 5 — Reseed + build verification

**Issue discovered:** `npm run seed --reseed-blogs all` finds existing records by slug. When a slug RENAMES, the old slug record is orphaned (script creates new record, doesn't delete old). DB ended up with both old + new slugs, causing duplicate routes in the build.

**Fix applied:** Created `_delete-orphan-slugs.mjs` (in this folder) — one-off Payload API script that deletes the 4 orphan records by old slug. After running it + reseeding, the build correctly:
- Generated `/blog/[slug]` for 15 routes (was 19 before — 4 moved out)
- Generated `/[citySlug]` for 112 routes (90 cities + 22 legacy-root blog posts including the 4 newly restored)

**Files in this folder:**
- `_delete-orphan-slugs.mjs` — the orphan cleanup script (keep for future reseeds when slugs change)

## Standing-rule guidance for future sessions

**When a slug RENAMES in `blog-data.ts`:** the seed script alone is insufficient. Must:
1. Delete the orphan DB record(s) for the old slug first (via `_delete-orphan-slugs.mjs` pattern in this folder)
2. THEN run `npm run seed --reseed-blogs all`

This is a known seed-script limitation. Worth fixing in seed.ts long-term — should detect slug-only changes and update existing records or delete orphans automatically.
