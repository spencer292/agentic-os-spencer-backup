## Optimization — What Was Done vs Notion Plan

The prior Notion plan proposed uploading originals to Vercel Blob and letting the CDN resize on the fly. That decision was reversed in the 2026-04-14 session: **pre-optimize locally, upload only the optimized WebP files**. This doc reflects the local pipeline that actually ran.

### Ran this session (2026-04-15)
| Tier | Max dimension | Quality | Total size | Avg per image | Matches site precedent |
|------|--------------|---------|-----------|---------------|----------------------|
| **Hero** | 1920px long side | 82 | 28.0 MB | ~290 KB | `public/images/hero-*.webp` (1920×1072) |
| **Content** | 1200px long side | 82 | 14.3 MB | ~150 KB | `public/images/blog-*.webp` (1200×669) |

Originals preserved untouched in `brand_context/pictures/`. Optimized files live in `brand_context/pictures/optimized/{hero,content}/`.

### Next: Vercel Blob upload
Upload the optimized WebP files (not originals) to Vercel Blob. Next.js `<Image>` component handles further srcset generation and browser-specific format selection (WebP/AVIF) at request time.

### Performance budget
Target: every page loads under 3 seconds on 3G. Total image weight per page stays under 500 KB. With lazy loading, only above-fold images count toward initial load — typically 1 hero + 1–2 inline = ~300 KB max. Hero tier at 290 KB avg fits this budget cleanly.

### Usage pattern in code
The Hero block (`src/blocks/Hero.ts`) has two image fields:
- `backgroundImage` — CMS upload relationship (unused so far, reserved for Payload Media / Vercel Blob route)
- `fallbackImage` — string key, resolves to `/images/{key}.webp`

For these 20 photos, set `fallbackImage` in `src/lib/pages-data.ts` using the filename without `.webp`. Example:
```ts
heroBlock: {
  fallbackImage: 'hero-team-line',  // resolves to /images/hero-team-line.webp
  ...
}
```

Team headshot cards on the About page will need a separate content block that references the `team-{name}` files directly (or via a new `TeamCards` block — check whether one already exists in `src/blocks/`).
